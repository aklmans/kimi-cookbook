<T zh="Kimi 的输入框不只是个打字框 —— 旁边有一个我自己长期没怎么动过的模式开关: Instant、Thinking、Agent、Agent Swarm。多数人一直留在默认那档, 没意识到换一档, 它干的活就完全是另一回事。" />

<T zh={<>这四个模式, 是 Kimi 把「答你一句」和「替你做成」分开的旋钮。<Footnote n={1} />挑错了, 要么简单问题让你干等, 要么把一句话能答的事支起一整套 agent。这章我给一张对照加四个场景, 以及一条贯穿始终的判断 —— 挑能办成事的<strong>最低</strong>那一档。</>} />

<SectionTitle number="I"><T zh="四个模式, 是从「答」到「做」的四档" /></SectionTitle>

<T zh="这四个模式排在一条阶梯上: 从「给你一句话」到「替你干一整件事」。越往右, 它替你扛得越多, 你付的时间、额度和风险也越大。" />

<Dl>
<Di term={<T zh="Instant · 快答" />} desc={<T zh="不开思考, 秒回, 几乎不耗什么。查事实、改错字、要立刻拿到一句话 —— 这一档够。" />} />
<Di term={<T zh="Thinking · 深想" />} desc={<T zh="先在内部推几步再答, 拿等待换深度。多步推理、算账、写码、要核对的活（思考在 K2.6 上可关, K3 与 K2.7-Code 永远开）。<Footnote n={3} />" />} />
<Di term={<T zh="Agent · 交办" />} desc={<T zh="不只给答案, 直接做出网站 / 幻灯 / 文档 / 表格。一跑就吃掉共享额度池里的一份（各档约合 6–360 个）和并发额度, 还得等它把成品做完。细看第五章。" />} />
<Di term={<T zh="Agent Swarm · 集群" />} desc={<T zh="派一群子 agent 并行干, 烧的是按档限次的集群额度（Moderato 25 次 / Allegretto 50 / Allegro 120, K3 驱动, 仍是 Beta）, 一次消耗约为普通 Agent 任务的数倍。<Footnote n={2} />细看第六章。" />} />
</Dl>

<T zh="一句话记住分界: 前两档（Instant / Thinking）是「它答、你做」, 后两档（Agent / Agent Swarm）是「它做、你验」。你这次到底要一段回答, 还是要一件成品, 决定了你该停在哪一档。" />

<SectionTitle number="II"><T zh="四个场景, 各停一档" /></SectionTitle>

<H3 id="scene-instant"><T zh="场景一 · 「这个 API 报错是什么意思」" /></H3>

<T zh={<>贴一段报错问原因 —— 要的是一句话。开 Instant, 秒回, 接着干活。<strong>这种活开 Thinking 是白等十几秒, 开 Agent 是等十分钟换一句本该秒回的话。</strong>日常问答, 默认就该停在这里。</>} />

<H3 id="scene-thinking"><T zh="场景二 · 「这笔账帮我核一遍」" /></H3>

<T zh={<>多步推理、容易算错一步就全错的活 —— 开 Thinking, 让它在内部先推几步。代价只是多等一点, 换来的是少犯低级错。<strong>边界: 它推它的, 你核你的; 深想提高正确率, 不豁免核对。</strong></>} />

<H3 id="scene-agent"><T zh="场景三 · 「把这周的数据做成汇报 PPT」" /></H3>

<T zh={<>要的是一件能直接交的成品, 不只是建议 —— 上 Agent。它会自己拆任务、做页面、导出文件, 你等几分钟到十几分钟, 烧掉共享额度池里的一份。<strong>出发前把话说清: 受众是谁、要几页、什么风格。交办模式最贵的不是额度, 是它把一个偏掉的方向一口气做到底。</strong>成品到手先验再发, 别直接转发。</>} />

<H3 id="scene-swarm"><T zh="场景四 · 「五十个网页, 给我一张竞品对照表」" /></H3>

<T zh={<>这活的特征是「多」不是「难」: 几十个网页要翻、十几份材料要读、一个问题要多个独立视角。单线干到天黑的事, 放一群子 agent 并行, 烧一次集群额度（Moderato 每月 25 次）。<Footnote n={2} /><strong>边界: 别为不存在的规模付费 —— 十个网页以内、一条线能读完的活, 一个 Agent 甚至 Thinking 就够, 动用 Swarm 纯属浪费次数。</strong></>} />

<T zh="我把四个场景收成一行判断: 先看活的形状, 再挑能办成它的最低那一档。" />

| 活的形状 | 停在哪档 | 实际代价 |
| --- | --- | --- |
| 要一句话 | Instant | 秒回, 几乎不耗什么 |
| 要一段讲得通的推理 | Thinking | 多等十几秒 |
| 要一件能交的成品 | Agent | 几分钟 + 共享额度一份 |
| 要并行一大批 / 多视角 | Agent Swarm | 集群次数一次（每月按档限量） |

<Divider />

<SectionTitle number="III"><T zh="默认够用, 别为「更高级」上更高的档" /></SectionTitle>

<T zh={<>一个常见的错觉是「上更高级的模式 = 更好的结果」。多数日常活, 停在前两档才对: 拿 Agent 去问一个事实, 是等十分钟换一句本该秒回的话; 拿 Swarm 去做一件单线就能办的事, 是为不存在的规模付费。<strong>把模式当工具挑, 不当「等级」追。</strong></>} />
