<SectionTitle number="I"><T zh="三颗在列的脑子, 各管一段" /></SectionTitle>

<T zh={<>重活归 <strong>kimi-k3</strong>: 2.8 万亿参数 MoE、1M 上下文、思考永远开(顶层 <C>reasoning_effort</C> 目前只有 <C>max</C> 一档, 关不掉)—— 长链路编码、百万 token 长料、反复推的知识工作归它, 它也最贵。<Footnote n={1} /><Footnote n={6} />编码专线归 <strong>kimi-k2.7-code</strong>: 1 万亿参数、256K, 思考强制开启; 高速变体 <C>kimi-k2.7-code-highspeed</C> 输出约 180 tokens/s、价格翻倍, 适合交互式结对。<Footnote n={2} /><Footnote n={5} />日常归 <strong>kimi-k2.6</strong>: 同样 256K、多模态, 思考可选可关, 还能走 Batch 批量(六折)—— K3 反而不在 Batch 内。<Footnote n={3} /><Footnote n={4} />至于 kimi-k2.5: 新用户已经用不了, 老用户也只能用到 2026 年 8 月 31 日, 别把新活再绑上去。<Footnote n={2} /></>} />

<T zh="K2.6 时代, 这章教你「活简单就关思考」。这个开关在 K2.6 上还在, 写法没变:" />

<CodeBlock caption={<T zh="关掉思考的最小调用(仅 kimi-k2.6 支持;K3 与 K2.7-Code 的思考关不掉, 传 disabled 会报错)" />}>

```python
from openai import OpenAI

client = OpenAI(api_key="YOUR_KIMI_API_KEY",
                base_url="https://api.moonshot.cn/v1")  # 与 OpenAI 兼容

resp = client.chat.completions.create(
    model="kimi-k2.6",
    messages=[{"role": "user", "content": "把这段会议纪要按议题归类, 列出待办"}],
    extra_body={"thinking": {"type": "disabled"}},  # 关思考: 活简单、要秒回时
)
print(resp.choices[0].message.content)
```

</CodeBlock>

<SectionTitle number="II"><T zh="任务成本解剖台" /></SectionTitle>

<T zh={<>下面四个任务, token 画像全摆出来, 价格用开放平台标准价逐笔算(每百万: K3 输入 ¥20 / 命中 ¥2 / 输出 ¥100;K2.7-Code 与 K2.6 输入 ¥6.5 / 命中 ¥1.1–1.3 / 输出 ¥27;高速版输入 ¥13 / 输出 ¥54)。<Footnote n={3} />你自己任务的账, 照这个格式套就行。</>} />

<H3 id="task-pdf"><T zh="任务一 · 通读 300 页 PDF, 出一份结构化纪要" /></H3>

<T zh={<>画像: 300 页中文约 20 万 token 输入, 纪要约 3 千 token 输出, 一次过。<strong>K3: 输入 20×0.2 + 输出 100×0.003 ≈ ¥4.3</strong>(追问时前缀走缓存, 再砍到 ¥0.7)。<strong>K2.6 / K2.7-Code: 6.5×0.2 + 27×0.003 ≈ ¥1.4</strong>。</>} />

<T zh={<>差价三倍, 但别急着选便宜的: 256K 的两颗塞 200K 材料只剩一成的余量, 提示词和输出再一挤, 要么砍料要么分段 —— 分段丢的是「贯穿三百页的主线」这类全局判断。结论很直接: <strong>10 万 token 以内的料, K2.6 上;逼近或超过 20 万, 这 ¥3 是买「一次吃完不丢全局」, 值。</strong></>} />

<H3 id="task-coding"><T zh="任务二 · 编码 agent 跑一个跨 10 文件的功能改动" /></H3>

<T zh={<>画像: agent 长链路, 20 轮迭代, 每轮平均 3 万 token 输入(仓库上下文, 前缀大量重复 —— 厂商称编码负载缓存命中率超九成, 按九成算)、2 千输出。合计输入 60 万(54 万命中 + 6 万未命中)、输出 4 万。<Footnote n={1} />按百万 token 计: <strong>K3: 0.06×20 + 0.54×2 + 0.04×100 ≈ ¥6.3;K2.7-Code: 0.06×6.5 + 0.54×1.3 + 0.04×27 ≈ ¥2.2;高速版同口径约 ¥4.3。</strong></>} />

<T zh={<>单看标价 K3 贵三倍, 但编码活要算的是「做对一次的成本」。独立 DeepSWE 榜(统一 harness): K3 修复率 69%, K2.7-Code 31% —— 把币值除以修复率, K3 约 ¥9 一次成功, K2.7-Code 约 ¥7。<Footnote n={7} />差距远没有标价悬殊, 而且还没算你盯着返工的时间。<strong>结论: 日常小改 K2.7-Code;一改错就要重来好几轮的难活, K3 买的是「一次过」。赶时间的结对场景换高速版 —— 翻倍的价格买的是你不等, 后台任务别碰它。</strong></>} />

<H3 id="task-batch"><T zh="任务三 · 一千条评论打标签(批量、可等)" /></H3>

<T zh={<>画像: 每条 500 输入 + 50 输出, 共 50 万输入、5 万输出, 走 Batch 六折(仅 K2.7-Code / K2.6 / K2.5)。<Footnote n={4} /><strong>K2.6 Batch: (6.5×0.5 + 27×0.05) × 0.6 ≈ ¥2.8, 单条不到 0.3 分钱。</strong>K3 不在 Batch 内, 标准价跑这批要 ¥15 —— 五倍多的钱, 买的是根本不会用的旗舰推理。</>} />

<T zh={<>打标签这种「答错一眼能看出来」的活, 顺手把 K2.6 的思考也关了, 又快又便宜。<strong>结论: 能批量的活永远先想 Batch + K2.6, 这是整张价目表上性价比最高的一格。</strong></>} />

<H3 id="task-chat"><T zh="任务四 · 日常问答与文档小修(高频、零碎)" /></H3>

<T zh={<>画像: 每次 2 千输入 + 5 百输出, 一天几十次。这种量级下三颗模型的单答成本都在一分钱上下 —— 价格在这里根本不构成决策变量。<strong>结论: 日常零碎活, 默认 K2.6 关思考;只有答案质量连续几次不满意, 才把这一类活升档。</strong>升档的触发器是「连续不满意」, 不是「万一这次难呢」。</>} />

<T zh={<>把四个任务收成一张表, 下次对着活直接找行:</>} />

| 10 万 token 内的摘要 / 翻译 / 归类 | kimi-k2.6(关思考) | 几毛 | 答错一眼能看出的活, 不配旗舰 |
| 20 万 token 级长料一次吃完 | kimi-k3 | 几元 | 分段会丢全局判断的活 |

<T zh={<>再看跑分 —— 先分两摞。<strong>第一摞, 厂商自评</strong>: K3 发布博客里的数字, 多数是官方用自家 harness 跑出、未经第三方榜单收录; 而官方在同一篇博客里写明, 整体表现仍「trails the most powerful proprietary models」, 点名 Claude Fable 5 和 GPT 5.6 Sol, 并在 Limitations 里承认与这两家有「a noticeable gap in user experience」, 还警告 K3 有过度自主倾向。<Footnote n={1} /><strong>第二摞, 独立实测</strong>: 发稿时刚开始落地 —— DeepSWE 官方榜(2026-07-17 更新, 全榜统一 mini-swe-agent harness)已收录 K3: 69%±5%、16 个模型第 4, 距榜首 73% 只差 4 个点、误差棒与二三位重叠, 单任务成本 $4.65 在前五里第二低; 同一张榜上 K2.7-Code 是 31%±1、第 14。SWE-bench Verified 则仍无 K3 条目, Kimi 最好成绩还是 K2.5 的 70.8%。<Footnote n={7} /><Footnote n={8} /><strong>第三摞, 偏好榜</strong>: Code Arena(WebDev)榜上 K3 以 1679 分登顶(约 48 万票)—— 但它是人类盲投的 Elo 偏好榜, 偏好不等于正确性。<Footnote n={9} /></>} />

<T zh={<>所以「什么时候回 frontier」这条线没变, 只是更好说了: 一锤定音的最难活(架构级的编码、多跳推理、不能错的输出)回 Claude / GPT —— 这不是我们替它谦虚, 是 Moonshot 自己写在发布博客里的。<Footnote n={1} />K3 值得付钱的场景同样清楚: 百万 token 一次吃完的长料、要跑很久的 agent 任务、国内直连的顺手, 以及(7 月 27 日之后)可自托管的开放权重。中间的日常活, K2.6 依旧够用。</>} />
